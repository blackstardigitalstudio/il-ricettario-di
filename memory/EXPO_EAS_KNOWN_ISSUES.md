## 📌 Checklist pre-build (da fare SEMPRE prima del primo `eas build`)

- [ ] Node.js ≥ 20 LTS installato sul PC (`node -v`)
- [ ] `yarn` globale installato (`npm install -g yarn`)
- [ ] **NESSUN** `package-lock.json` nel progetto (solo `yarn.lock`)
- [ ] Tutti i config plugin di librerie native verificati (vedi Problema #1)
- [ ] `eas.json` ha profili `development` / `preview` / `production` corretti
- [ ] Icone e splash screen presenti in `assets/images/`
- [ ] `app.json` → `android.package` e `ios.bundleIdentifier` settati
- [ ] `app.json` → `android.versionCode` e `ios.buildNumber` bumped
- [ ] Privacy Policy URL pronta (obbligatoria se usi AdMob / tracking)

## ⚠️ Problema #1 — Config plugin di `react-native-google-mobile-ads` non risolto

### Sintomo
```
Unable to resolve a valid config plugin for react-native-google-mobile-ads.
• No "app.plugin.js" file found in react-native-google-mobile-ads
• Unexpected token 'typeof' when importing lib/commonjs/index.js
```

### Causa
Il file `app.plugin.js` del pacchetto `react-native-google-mobile-ads` può mancare / essere bloccato / non raggiungibile su alcune installazioni Windows (antivirus, npm parziale, ecc.).

### Soluzione: plugin locale nel progetto
Crea `frontend/admob-plugin.js`:

```js
const { withAndroidManifest, withInfoPlist } = require('@expo/config-plugins');
const ANDROID_META_NAME = 'com.google.android.gms.ads.APPLICATION_ID';

function withAdMobAndroid(config, { androidAppId }) {
  if (!androidAppId) return config;
  return withAndroidManifest(config, (cfg) => {
    const app = cfg.modResults.manifest.application?.[0];
    if (!app) return cfg;
    app['meta-data'] = app['meta-data'] || [];
    const existing = app['meta-data'].find(
      (m) => m.$?.['android:name'] === ANDROID_META_NAME,
    );
    if (existing) {
      existing.$['android:value'] = androidAppId;
    } else {
      app['meta-data'].push({
        $: { 'android:name': ANDROID_META_NAME, 'android:value': androidAppId },
      });
    }
    return cfg;
  });
}

function withAdMobIos(config, { iosAppId }) {
  if (!iosAppId) return config;
  return withInfoPlist(config, (cfg) => {
    cfg.modResults.GADApplicationIdentifier = iosAppId;
    return cfg;
  });
}

module.exports = function withAdMob(config, props = {}) {
  config = withAdMobAndroid(config, props);
  config = withAdMobIos(config, props);
  return config;
};
```

E in `app.json`:
```json
"plugins": [
  ["./admob-plugin", { "androidAppId": "ca-app-pub-XXXX~YYYY", "iosAppId": "ca-app-pub-XXXX~ZZZZ" }]
]
```

## ⚠️ Problema #2 — `package-lock.json` e `yarn.lock` coesistono

### Sintomo durante `eas build` (phase: Install dependencies)
```
warning package-lock.json found. Your project contains lock files generated by tools other than Yarn.
error Your lockfile needs to be updated, but yarn was run with `--frozen-lockfile`.
yarn install --frozen-lockfile --production false exited with non-zero code: 1
```

### Causa
Expo EAS usa **yarn** internamente con `--frozen-lockfile`. Se qualcuno ha fatto `npm install` in locale, npm crea un `package-lock.json` che entra in conflitto con il `yarn.lock` esistente (che nel frattempo è diventato obsoleto perché npm non lo aggiorna).

### Soluzione
Sul PC sviluppatore:
```bash
cd frontend
del package-lock.json      # Windows
# rm package-lock.json     # macOS/Linux
rmdir /s /q node_modules   # Windows
# rm -rf node_modules      # macOS/Linux
yarn install               # rigenera yarn.lock corretto
git add yarn.lock
git commit -m "refresh yarn.lock"
git push
```

### Prevenzione
Aggiungi nel `.gitignore`:
```
package-lock.json
```

E nel README o nelle istruzioni dev scrivere **"USA SOLO YARN"** in grande.

Aggiungi anche questo script di guardia in `package.json`:
```json
"scripts": {
  "preinstall": "npx only-allow yarn"
}
```
(blocca chi prova a fare `npm install` e lancia un errore chiaro).

## ⚠️ Problema #3 — `Run this command inside a project directory`

### Sintomo
```
eas build:configure
Run this command inside a project directory.
Error: build:configure command failed.
```

### Causa
L'utente sta eseguendo `eas` dalla cartella radice del monorepo (`il-ricettario-di/`) invece che dentro la cartella Expo vera (`frontend/`). EAS cerca `app.json` e `package.json` nella cartella corrente.

### Soluzione
**Sempre** `cd frontend` prima di qualsiasi comando `eas` o `expo`.

### Prevenzione
Crea uno script batch `build-android.bat` nella radice:
```batch
@echo off
cd frontend
eas build --platform android --profile production
```

## ⚠️ Problema #4 — `npm install` fallisce con ENOENT su `package.json`

### Sintomo
```
npm error code ENOENT
npm error enoent Could not read package.json
```

### Causa
Stesso del Problema #3: si sta lanciando `npm install` dalla cartella che non contiene `package.json`.

### Soluzione
`cd frontend` prima.

## ⚠️ Problema #5 — Versione Node.js troppo vecchia

### Sintomo
- `Unexpected token` durante il parsing di JS compilato
- Pacchetti che falliscono a installarsi
- `optional chaining` non riconosciuto

### Soluzione
- Richiedi **Node.js 20+ LTS** (preferibile 22)
- Documenta in README: `"engines": { "node": ">=20" }` in `package.json`

## ⚠️ Problema #6 — `Detected that your app uses Expo Go for development`

### Sintomo (warning non bloccante)
```
⚠ Detected that your app uses Expo Go for development, this is not recommended when building production apps.
```

### Causa
Non è un errore — è solo un warning che ti ricorda che in produzione non puoi usare Expo Go per feature native (AdMob, Share, Push, ecc.).

### Silenziare
```bash
set EAS_BUILD_NO_EXPO_GO_WARNING=true
```

## ⚠️ Problema #7 — `keytool non installed locally`

### Sintomo (info, non errore)
```
Detected that you do not have keytool installed locally.
Generating keystore in the cloud...
```

### Nota
Va bene così. EAS genera il keystore sul cloud Expo e lo gestisce per te. Puoi sempre scaricarlo in qualsiasi momento con:
```bash
eas credentials
```
**IMPORTANTE**: fai backup del keystore. Se lo perdi, non puoi più aggiornare l'app sul Play Store (dovresti creare un nuovo pacchetto).

## ⚠️ Problema #8 — Config plugin di una libreria che richiede source compilation

### Sintomo generico
```
Unable to resolve a valid config plugin for <pacchetto>.
```

### Librerie storicamente problematiche
- `react-native-google-mobile-ads` (vedi Problema #1)
- `react-native-firebase/*` (richiede config plugin corretto)
- `react-native-permissions` (potrebbe servire plugin custom)
- `@sentry/react-native` (richiede pre-build hook)

### Soluzione generale
Se un config plugin non funziona:
1. Controlla che il pacchetto abbia `app.plugin.js` (`ls node_modules/<pkg>/app.plugin.js`)
2. Se manca o è rotto → scrivi un plugin locale usando `@expo/config-plugins`
3. Modifica `app.json` per puntare al plugin locale (`"./my-plugin"` invece del nome del pacchetto)

## ⚠️ Problema #9 — Versione libreria incompatibile con Expo SDK

### Sintomo
```
warning <pacchetto>@X.Y.Z has unmet peer dependency expo@~54.0.0
```
oppure build fallisce durante "Run gradlew" con errori Kotlin/Java.

### Soluzione
Usa SEMPRE `npx expo install <pacchetto>` invece di `yarn add <pacchetto>`. Questo prende la versione compatibile con la tua versione di Expo SDK.

Controlla le compatibilità correnti:
```bash
npx expo-doctor
```

## ⚠️ Problema #10 — Permessi Android non dichiarati

### Sintomo
App rifiutata da Google Play Review o crasha al lancio.

### Soluzione
In `app.json`:
```json
"android": {
  "permissions": ["INTERNET", "ACCESS_NETWORK_STATE"],
  "blockedPermissions": [
    "android.permission.WRITE_EXTERNAL_STORAGE",
    "android.permission.READ_EXTERNAL_STORAGE"
  ]
}
```
- Dichiara solo i permessi usati davvero
- Blocca esplicitamente quelli richiesti da librerie ma non necessari (evita rejection)

## 📝 Ordine di attacco quando il build fallisce

1. **Leggi attentamente il messaggio**: la phase che fallisce (Install deps / Prebuild / Gradle / Xcode) ti dice dove cercare.
2. **Apri il link `See logs:`** nella dashboard Expo — la UI web è molto più leggibile del terminale.
3. **Se fallisce in "Install dependencies"**:
   - Controlla conflitto lockfile (Problema #2)
   - Controlla peer deps con `npx expo-doctor`
4. **Se fallisce in "Prebuild" / "Install Pods"**:
   - Controlla config plugin (Problemi #1, #8)
5. **Se fallisce in "Run gradlew" (Android)**:
   - Controlla versioni nativi con `npx expo install --check`
   - Problemi di Kotlin version → guarda `expo-build-properties`
6. **Se fallisce in "Run fastlane" (iOS)**:
   - Controlla bundle identifier, certificati, provisioning profiles
